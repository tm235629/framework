## Task Routing — classify the request BEFORE reading anything else

The Drive serves several distinct workflows. **Pick the narrowest matching row, read only what it
prescribes.** The default is silence — do not speculatively load files, indexes, or tools. The rows below
are **generic archetypes**; their *examples* draw on `{company-slot}` paths but the routing logic is invariant.

| Request archetype | What it looks like | What to read | What NOT to read |
|---|---|---|---|
| **Targeted file edit** | "rename X to Y in `<path>`", "fix the typo in `<file>`", "add a comment in `<script>`" | The target file. Its sibling rule-file (`{rule-filename}`) **only** if behavioral rules might apply. | Anything else. No `{state-file}`, no other folders, no tools. |
| **Focused folder / project work** | "help with the `<entity>` design", "fix the `<entity>` script", "work in `<folder>`" | The folder's `{rule-filename}` + its `{overview-filename}` TL;DR (`Read --limit 30`). The specific files named. | Other projects, standards, trackers, tools, the project index. |
| **Status / lookup question** | "when did we last `<recurring-action>`?", "what's the next `<entity>` milestone?" | `{state-file}` only. `Grep` for a single fact from a flagged file. | Standards, project overviews, multiple files. |
| **Cross-entity question** | "which `<partners>` are dormant?", "what links `<entity-A>` to `<entity-B>`?" | The structured index CLI (`{graph-cli}`) if it saves reading; else grep TL;DRs. | Reading every `{overview-filename}`. |
| **Workflow trigger** | any `{triggers}` phrase (e.g. `{sync-trigger}`, `{ingest-trigger}`) | Invoke the named skill — it loads what it needs. | Standards or trackers ahead of the skill — it defers them to specific steps. |
| **External-facing copy** | website, brochure, whitepaper, press release, investor copy, bios | `{style-guide}` first; then the draft + the relevant entity's `{overview-filename}` (fact-check every claim). | `{state-file}`, trackers, internal syncs, unrelated overviews. |
| **Ambiguous / open-ended** | "what's going on with `<entity>` lately?", "give me a summary" | Take the most reasonable reading, **state the assumption you're running with**, and answer it. Ask only when genuinely-different meanings change the work. | The large mutable trackers or every overview at once. |

**The convention is: targeted by default, broad by exception.** With a large context window, reading an
extra overview or standard when it clearly helps is cheap — read it and note that you did. Stay disciplined
only about the large mutable files flagged `{avoid-read-marker}`: reach for `{state-file}` or an extractor
first. Otherwise don't pause for permission to read — widen scope when it helps, and proceed.

## Avoid-read convention

Files carrying `{avoid-read-marker.flag_field}: {avoid-read-marker.flag_value}` in frontmatter are large
(>15–25 KB) and their headline state is summarized elsewhere. **Prefer `{state-file}`, the file's own TL;DR
head (`Read --limit 30`), or the script named in `{avoid-read-marker.extractor_field}`.** A "full read" —
loading the *entire* file into context — is what the convention discourages. Targeted access
(`Read --offset/--limit`, `Grep`, an extractor's JSON output) is always fine.

**Override:** advisory, not a block. Read the full file when (a) the user explicitly asks, (b) the extractor
or TL;DR doesn't surface what you need, or (c) you've narrowed the need but the file lacks structure for a
targeted read. **Note in your reply when you bypass it** so the user can correct you.

**What's flagged today:** grep `{avoid-read-marker.flag_field}: {avoid-read-marker.flag_value}` for the live
list — `{avoid-read-list}` *(slot: the instance's current set of flagged trackers/overviews; do not hardcode —
it grows)*.

## Superseded / legacy convention *(applies to every file, not just flagged ones)*

If a file carries `status: {superseded}` or `status: {legacy}` frontmatter (members of `{status-enum}`),
sits in a `{superseded-folder}` sibling, or opens with a `⚠️ SUPERSEDED` banner: **read only the banner +
frontmatter and follow the `superseded_by` / decision link to the current document** — do not treat its
content as current. This matters most after a `Grep` hit: **check the first 30 lines before using a match.**
Exception: when the user explicitly asks about history ("why did we abandon X?"), the full superseded doc is
the right source.

> Slots: `{superseded}`/`{legacy}` = the aging members of `company_profile.vocab.status_enum`;
> `{superseded-folder}` = the category-rule that buckets a path to `superseded`
> (`taxonomy.category_rules` where `category === "superseded"`).

## General rules

- **File changes follow three autonomy tiers.** Route every change to the lowest applicable tier:
  - **(1) Annotate / curate** — additive, reversible lifecycle metadata (`status`/`context` frontmatter,
    supersession edges, banners, decision entries, catalog & content summaries, current-position blocks):
    **autonomous — just note it in your summary.**
  - **(2) Restructure** — moves, renames-to-convention, archiving a superseded version into a
    `{superseded-folder}` sibling: **autonomous but announce every move.**
  - **(3) Destroy / rewrite** — deleting a file, overwriting a document's substance, or modifying content you
    did not create: **confirm with the user first.**
  > Full spec: `{lifecycle-standard}`. The tier boundaries are invariant; the cited standard is a `{company-slot}`.
- **Never permanently delete** from `{raw-archive-roots}`. Move junk to a `cleanup/` sibling instead.
- **Action items / to-dos:** do NOT assign a name unless the source makes responsibility explicit. Do NOT add
  priority tags — the team handles prioritization.
- **Entity folders** *(`taxonomy.project_tiers[].entity_card === true`)*: one folder per entity, named per the
  tier's `naming_convention`; all subsequent materials go in the same folder — **never** a duplicate folder for
  the same entity. **Location ≠ owner:** a folder lives where its artifacts *originate*; the owning account is
  expressed with a typed `references` edge (`{edge-vocab}`, e.g. `customer`), not by moving the folder.
- **Graph registration:** every new `.md` must carry YAML frontmatter with at minimum `{required-fields}`.
  `references` is for **genuine cross-tree links only** — **never** an ancestor/containment-line directory
  (that edge is already drawn). No genuine cross-ref yet → `references: None` (keep the field). Use only the
  valid `{edge-vocab}` types; never `parent`/`child`. Define each edge once.
- **Catalog maintenance:** after adding/removing/moving files, refresh `{catalog-name}` files with
  `{catalog-tool}` (dry-run first). Prefer incremental over force-rewrite to avoid synced-cloud churn.
- **Session learnings:** at the end of a substantive session, record 2–3 concise, reusable, non-obvious
  lessons in `{learnings-location}` — one entry per genuine lesson.

> Slots: `{lifecycle-standard}`, `{learnings-location}` = the instance's governance docs; `{catalog-name}` =
> `catalog_profile.catalog_filename`; `{catalog-tool}` = the instance's walker; `{raw-archive-roots}` =
> `company_profile.raw_archive_roots`.

## Optional CLI tools — `{cli-tools}`

These are **opt-in** — load them only when they save reading or grep for the current task. For targeted edits
or focused folder work, do not invoke them at all. They emit structured output (JSON / short text) so a small
call replaces multiple reads.

*Slot: the instance's B-library exposed as commands — the structured-index/graph tool (`{graph-cli}`), the
state generator (`{state-generator}`), the TL;DR extractor, the project-filter query, the data-layer extractor,
the catalog walker (`{catalog-tool}`, a **write** tool — not for reading), the PDF renderer, the dashboard
server. One row each: tool · when-to-use · command. Each is a pure function of the manifest — no per-instance
code edit, only a different `manifest.json`.*

## Project Index — `{project-index}`

Each entity has an `{overview-filename}` in its folder; the TL;DR block at the top (`Read --limit 30` or the
extractor) gives status, contacts, next milestone — **so live status can't go stale in this hub.**

**Tiering model.** Every entity `{overview-filename}` carries three filterable/sortable frontmatter fields:

- **`tier`** — *strategic importance* (`{tier-scale}`: `{tier-scale.min}` = focus … `{tier-scale.max}` = minor),
  **independent of current activity** (a dormant marquee keeps its tier).
- **`phase`** — *engagement state*: one of `{phase-enum}`.
- **`vertical`** — *product lane*: one of `{verticals}`.

The **authoritative, complete tiering lives in the frontmatter** (and the dashboard that reads it). The lists
below are a **navigation shortcut to the most-touched entities per tier — not exhaustive.**

*Slot: render `entity_registry` cards grouped by `tier`. For each tier (`{tier-scale}` label as the heading),
a short table of the anchor entities: name + `{overview-filename}` path. Only `entity_card:true` tiers appear;
`non_card_subfolders` working zones never become tiles. Filled by the card extractor, not hand-maintained.*
